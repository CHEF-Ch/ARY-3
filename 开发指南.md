### 后端约定

**数据层**（`server/src/db.ts`）：

```ts
import { v4 as uuid } from "uuid";
import { findAll, findById, findBy, filterBy, insert, update, remove } from "../db.js";

// 举例
const user = findById("users", userId);
const races = filterBy("races", "status", "running");
insert("works", { id: uuid(), title: "作品名", ... });
update("users", userId, { displayName: "新名字" });
```

每个"表"是 `database/data/` 下的一个 JSON 文件，结构是对象数组。服务启动时自动创建。

**鉴权**（`server/src/shared/auth.ts`）：

```ts
import { authorize, getCurrentUser, requireLogin } from "../shared/auth.js";

// 路由中使用
router.post("/races", requireLogin, (req, res) => {
  const user = getCurrentUser(req);
  const result = authorize(user, "Race", "create", { visibility: "public" });
  if (!result.allowed) return res.status(403).json({ error: result.reason });
  // 正常业务逻辑
});
```

**authorize 的四个作用域**：

| 作用域 | 含义 | ctx 参数 |
|--------|------|---------|
| PUBLIC | 无需登录 | `{ visibility: "public", isPublished: true }` |
| OWN | 资源所有者 | `{ ownerUserId: resource.user_id }` |
| ASSIGNED | 分配给当前评委 | `{ assignedJudgeUserId: judgeId }` |
| MANAGED_RACE | 赛事主办方 | `{ raceOrganizerIds: [org1, org2] }` |

**权限矩阵**已写在 `auth.ts` 里——13 种资源类型 × 每个动作对应的作用域。你只需要调 `authorize()`，不需要自己判断角色。

**路由注册**：

```ts
// 在 server/src/modules/race-mgmt/routes.ts 中
import type { Express } from "express";
import { Router } from "express";

export function registerRaceMgmtRoutes(app: Express): void {
  const router = Router();
  router.get("/", (req, res) => { ... });    // → /races/
  router.get("/:slug", (req, res) => { ... }); // → /races/:slug
  app.use("/races", router);
  console.log("[race-mgmt] Routes registered: /races/*");
}
```

**在 app.ts 中激活**（A 已经做好了 identity 和 communication 的示范）：

```ts
// server/src/app.ts — 取消注释你的模块即可
import { registerRaceMgmtRoutes } from "./modules/race-mgmt/routes.js";
registerRaceMgmtRoutes(app);
```

### 前端约定

- Vite + React + React Router
- `client/vite.config.ts` 已将 `/auth`、`/races`、`/works` 等路径代理到 `http://localhost:3000`
- 页面在 `client/src/pages/<页面名>/` 下，导出默认组件
- 路由在 `client/src/App.tsx` 中定义，需要新增路由时加一行 `<Route>`

## 二、如何在框架下工作

### 启动项目

```bash
# 终端 1：后端
cd server
npm install
npm run dev          # → http://localhost:3000

# 终端 2：前端
cd client
npm install
npm run dev          # → http://localhost:5173
```

### 工作流程（每人一样）

```
1. git clone <团队仓库>
2. git checkout -b feat/<你的模块名>
3. 在 server/src/modules/<你的模块>/ 下写 routes.ts
4. 在 client/src/pages/<你的页面>/ 下写页面组件
5. 需要新"表"时，在 database/migrations/ 下加 .sql 文件（记录字段），
   实际数据用 db.ts 的 JSON store 操作
6. npm run dev 本地自测
7. git push 提 PR

> **注意**：`database/migrations/*.sql` 仅作字段参考，不需要用 SQL 工具执行。实际数据全部通过 `db.ts` 的 JSON store 读写。
```

### 跨模块调用

你需要读其他模块的数据时，直接调 `db.ts`：

```ts
// C 在 portfolio 模块中需要读 race-mgmt 的数据
import { findById } from "../../db.js";
const registration = findById("registrations", work.registration_id);
```

**不要** import 其他模块的 routes.ts。数据层是共享的，路由层是隔离的。

### 鉴权

每个需要登录的接口，第一行写：

```ts
router.post("/xxx", requireLogin, (req, res) => {
  const user = getCurrentUser(req);
  const result = authorize(user, "资源类型", "动作", { ownerUserId: ... });
  if (!result.allowed) return res.status(403).json({ error: result.reason });
  // ...
});
```

不需要登录的公开接口不加 `requireLogin`。

### B — race-mgmt

| 交付物 | 目标 |
|--------|------|
| races 表 + CRUD 接口 | `POST /races`、`GET /races`、`GET /races/:slug`、`PATCH /races/:id`、`POST /races/:id/publish`、`POST /races/:id/archive` |
| registrations 表 + 接口 | `POST /races/:id/registrations` 报名、`POST /registrations/:id/approve` 审核通过（触发 RaceProject 幂等生成）、`POST /registrations/:id/reject`、`POST /registrations/:id/withdraw` |
| race_projects 表 + 接口 | approved 后自动创建，`GET /race-projects/:id` 查看状态 |
| ca_connections 表 + 接口 | `POST /race-projects/:id/ca-connections` 登记连接、`PATCH /ca-connections/:id` 管理元数据 |
| sessions 表 + DCR 对接 | `POST /ca-connections/:id/sessions/push` 接收 DCR push 消息（HMAC 验签 + nonce 防重放）、未登记/未握手数据拒收并隔离审计 |
| 聚合状态派生 | RaceProject.aggregate_ingestion_status 从 CAConnections 派生（not_configured→connected→active→failed） |
| 前端 | Race Page（按状态切换 CTA）、Rider View（参赛工作台）、Organizer View 中的报名/CA 状态部分 |

**关键不变量**：

- 一个 User 对同一 Race 最多一个 Registration（UNIQUE(race_id, user_id)）
- Registration approved 后幂等生成 RaceProject（重复审批不产生多个）
- CA 接入 failed 不触发 Registration 退赛，只生成评审前风险提示

### C — portfolio

| 交付物 | 目标 |
|--------|------|
| works 表 + CRUD 接口 | `POST /works` 创建、`PATCH /works/:id` 编辑、`POST /works/:id/submit` 提交、`POST /works/:id/lock` 锁定、`POST /works/:id/publish` 发布 |

- 一个 Registration 最多一个主 Work（UNIQUE(registration_id)）
- Award.race_id 必须与 Registration.race_id 一致
- Award 发布前不对 Public 可见

### D — projection

| 交付物 | 目标 |
|--------|------|
| 8 种 Projection 生成 | race_progress / registration_status / cost / risk / submission / judging / current_leaderboard / screen_feed |
| Projection 重建接口 | `POST /projections/:raceId/rebuild` Organizer 手动触发，失败保留最近稳定版本 |
| projections 查询接口 | `GET /projections/:raceId/:type` 公开读取 |
| Read Models | `leaderboard_read_model`、`results_page_read_model`、`review_summary_read_model`，分别服务最终榜单、Results 页和 Review 页读取 |
| fallback 语义 | 读取接口显式区分 `live` / `stable_fallback` / `static_fallback`，并返回结构化 `fallback` 元信息 |
| 状态摘要接口 | `GET /projections/:raceId/status-summary`，供 Organizer / 内部维护查看 projection/read model 可服务性、fallback 状态与最近成功时间 |
| 前端接入 | Live Hall、Screen Console、Screen Display、Home 已接入 projection；Results / Review 已接入 D 提供的 read model，页面归属仍分别属于 C / E |

**关键约束**：

- Projection 只读消费 race-mgmt 和 portfolio 的数据，从不回写
- 失败不污染上游核心事实表
- `current_leaderboard` 仅表示过程榜，不得作为最终结果事实源
- 最终榜单只来自 `Award + leaderboard_read_model`
- 原始 CA Session 默认不对公开端暴露，公开端只读取 projection / read model 摘要
- `screen_feed` 的 data 必须包含 `feedItemType` 字段区分过程榜和最终榜
- 所有 projection / read model 响应统一返回 `contractVersion`、`readKind`、`sourceOfTruth`、`consumerBoundary` 与结构化 `fallback`

**当前完成口径**：

- D 第一轮核心实现已完成，包含 projection 路由、`013-projections.sql`、fallback、read model、状态摘要与模块回归脚本
- `server/src/modules/projection/test.sh` 已覆盖公开/私有读取、Organizer 重建、过程榜/最终榜边界、fallback、契约元信息与状态摘要断言

**继续优化方向**：

- `screen_feed` 空模式替补策略与轮播协议继续细化
- `current_leaderboard`、`risk`、`event_stream` 的解释性与排序口径继续收敛

## 四、测试

### 模块测试（每人独立完成）

每个人在自己的模块目录下放一个 `test.sh` 文件（Windows 用 Git Bash 执行）。

**测试原则**：
- 只测自己的接口，不调别人的模块
- 依赖其他模块的数据时，直接用 `db.ts` 写入假数据
- 测试完成后清理假数据

**示例：race-mgmt 的 test.sh**（B 创建）

```bash

BASE="http://localhost:3000"

# 1. 先创建假用户（模拟 identity 的产出）
echo "--- 准备假用户 ---"
curl -s -X POST $BASE/auth/github \
  -H "Content-Type: application/json" \
  -d '{"githubAccount":"test-rider","displayName":"测试选手"}' -c /tmp/ary-cookie.txt
echo ""

# 2. 创建赛事
echo "--- 创建赛事 ---"
curl -s -X POST $BASE/races \
  -H "Content-Type: application/json" \
  -b /tmp/ary-cookie.txt \
  -d '{"title":"测试赛","challenge":"赛题描述","slug":"test-race"}'
echo ""

# 3. 报名
echo "--- 报名 ---"
curl -s -X POST $BASE/races/test-race/registrations \
  -H "Content-Type: application/json" \
  -b /tmp/ary-cookie.txt \
  -d '{}'
echo ""

# 4. 审核通过 → 应自动生成 RaceProject
echo "--- 审核通过 ---"
curl -s -X POST $BASE/registrations/<reg-id>/approve \
  -H "Content-Type: application/json" \
  -b /tmp/ary-cookie.txt
echo ""

echo "所有测试完成"
```
